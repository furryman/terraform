# Design: Upgrade k3s cluster to Kubernetes 1.36, with fully declarative bootstrap

**Status:** Approved design, awaiting implementation plan.
**Date:** 2026-05-24
**Author:** Adam Fuhriman (with Claude)
**Scope:** AWS-side infrastructure (`terraform/`) and Helm charts (`eks-helm-charts/`).
**Sibling repo impact:** `eks-helm-charts` (chart bumps), `argocd-app-of-apps` (no changes expected), `fuhriman-website` (no changes expected).

---

## Context

The single-node k3s cluster on `t4g.medium` is currently running **k3s v1.30.4+k3s1** (Kubernetes 1.30), 6 minor versions behind the latest stable **k3s v1.36.1+k3s1** (Kubernetes 1.36). All in-cluster workloads are managed declaratively by ArgoCD via the App-of-Apps pattern, so cluster state is fully reconstructible from git.

The current bootstrap is split between Packer-baked AMI content and a ~55-line `user_data.sh` shell script that runs at cloud-init. The script does three things at runtime:

1. Fetches the instance's public IP from IMDSv2 and re-runs the k3s installer to add it as a TLS SAN.
2. `helm install`s ArgoCD.
3. `helm install`s `argocd-apps` to bootstrap the App-of-Apps Application.

Inspecting the live API server cert reveals the public-IP TLS SAN is **unused** â€” port 6443 is not exposed (CLAUDE.md: "no public k8s API"), and the kubeconfig pins `server: https://127.0.0.1:6443` accessed exclusively via SSM tunnel. Removing the public-IP SAN removes the only "must run at boot" constraint, allowing the bootstrap to move entirely to Packer time via k3s's built-in helm-controller and auto-deploying manifests (`/var/lib/rancher/k3s/server/manifests/`).

Cert budget check (2026-05-24): only 1 Let's Encrypt issuance in the past 41h vs the 5-per-168h duplicate-certificate limit. The AMI rebuild strategy is safe from rate-limit considerations for this window.

## Goals

1. Bring the cluster to Kubernetes 1.36 (k3s v1.36.1+k3s1).
2. Bring all Helm chart dependencies (`cert-manager`, `envoy-gateway`, `external-dns`) to their current latest stable versions in the same change.
3. **Eliminate `user_data.sh` in favor of fully declarative bootstrap baked into the Packer AMI** â€” k3s server config, ArgoCD HelmChart CR, and App-of-Apps Application all live in `/var/lib/rancher/k3s/server/manifests/`.
4. Preserve public availability of `fuhriman.org`, `www.fuhriman.org`, `argocd.fuhriman.org`. Downtime budget: **~10 minutes** during instance replacement.
5. Preserve git as the source of truth â€” no in-cluster mutations, no `kubectl edit` workarounds.

## Non-goals

- Changing the cluster topology (still single-node, single-AZ, t4g.medium).
- Migrating away from k3s, Packer, or the App-of-Apps pattern.
- Bumping `fuhriman-chart` (in-house chart, already at 1.0.0 and unaffected by k8s version).
- Bumping the ArgoCD Helm chart itself â€” the live install at `9.5.15` is already the latest stable.
- Re-exposing the k8s API publicly. SSM tunnel remains the only admin path.

## Decisions

### D1: Target k8s minor version â†’ **1.36 (k3s v1.36.1+k3s1)**

Chosen over: v1.35.5+k3s1 (one back, more baked), v1.30.x patch (zero-risk no-progress). Rationale: a 6-minor jump and a 5-minor jump cost the same in chart compatibility work; might as well land on current.

### D2: Upgrade strategy â†’ **AMI rebuild + instance replace**

Chosen over: in-place upgrade (~30s API blip, preserves state), blue/green (~$0.50 overlap cost, zero downtime). Rationale: the repo's plumbing (App-of-Apps fully reconstructible state, Packer pipeline already wired through GitHub Actions OIDC) is purpose-built for this strategy. Cumulative etcd migrations across 6 minors in an in-place upgrade is the largest risk to avoid; clean-slate sidesteps it entirely.

### D3: Chart bump scope â†’ **all charts to latest stable**

Chosen over: minimum-for-1.36-compat, minimum-now-sweep-later. Rationale: opportunistic maintenance sweep while we're in this state anyway. Avoids a follow-up cycle and gets every component to the same "modern" baseline.

### D4: Sequencing â†’ **chart-bump PR before k3s-bump PR**

Smoke-test chart bumps against the live 1.30 cluster first (most chart bumps are backward-compatible). Catches chart issues without confounding them with k3s issues. If chart drift appears, fix in isolation before the AMI rebuild.

### D5: Bootstrap â†’ **fully declarative via Packer-baked manifests**

Remove `user_data.sh` entirely. k3s reads `/etc/rancher/k3s/config.yaml` for server args, and the built-in helm-controller auto-applies any YAML in `/var/lib/rancher/k3s/server/manifests/` at startup â€” including k3s's native `HelmChart` CR, which makes `helm install` an in-cluster reconciliation loop rather than a shell command. Chosen over: cloud-config YAML (still imperative at boot), keep `user_data.sh` (status quo). Rationale: removes the unused public-IP TLS-SAN runtime fetch, removes the `user_data_replace_on_change = true` foot-gun, collapses two bootstrap mechanisms into one, and makes the AMI self-contained and testable in isolation.

### D6: Drop public-IP TLS SAN

The runtime IMDSv2 fetch + k3s installer re-invocation in `user_data.sh` exists solely to add the instance's public IP to the API server cert's SAN list. Inspection of the live cert confirms this SAN is unused â€” port 6443 is not internet-accessible, and the kubeconfig pins `127.0.0.1` (reached via SSM tunnel). Removing the SAN removes the runtime dependency. k3s's default SANs (127.0.0.1, localhost, node hostname, internal IP, kubernetes.default.svc, etc.) cover all real client paths.

### D7: Drop `argocd-apps` Helm chart in favor of a plain `Application` manifest

The `argocd-apps` chart's only purpose in `user_data.sh` was to render one ArgoCD `Application` resource (the App-of-Apps parent). With declarative bootstrap, we can place that Application YAML directly in `/var/lib/rancher/k3s/server/manifests/` â€” no chart needed, no version pin to maintain, no 1.6.2 â†’ 2.x major bump to validate.

## Architecture

### Before (current)

```text
Packer build â†’ AMI contains:
  - k3s binary + systemd unit (k3s.service)
  - helm CLI + cached argo repo
  - amazon-ssm-agent (disabled by default)

Instance boot:
  cloud-init runs user_data.sh:
    1. wait for network
    2. enable amazon-ssm-agent
    3. fetch public IP via IMDSv2
    4. re-run k3s installer with --tls-san=<public-ip>
    5. wait for k3s API ready
    6. helm install argocd
    7. helm install argocd-apps (renders App-of-Apps Application)
```

### After (proposed)

```text
Packer build â†’ AMI contains:
  - k3s binary + systemd unit (k3s.service, ENABLED)
  - amazon-ssm-agent (ENABLED via systemctl)
  - /etc/rancher/k3s/config.yaml (server args)
  - /var/lib/rancher/k3s/server/manifests/00-argocd-namespace.yaml
  - /var/lib/rancher/k3s/server/manifests/10-argocd-helm.yaml (HelmChart CR)
  - /var/lib/rancher/k3s/server/manifests/20-app-of-apps.yaml (ArgoCD Application)

Instance boot:
  systemd starts k3s.service
    â””â”€â”€ k3s reads /etc/rancher/k3s/config.yaml
    â””â”€â”€ k3s starts API server + helm-controller
    â””â”€â”€ helm-controller applies manifests from /var/lib/rancher/k3s/server/manifests/
          â””â”€â”€ argocd namespace created
          â””â”€â”€ HelmChart CR triggers in-cluster helm install of argo/argo-cd 9.5.15
          â””â”€â”€ App-of-Apps Application created â†’ ArgoCD reconciles children
```

### Files modified in `terraform/`

- `packer/k3s-portfolio.pkr.hcl` â€” add `file` provisioners to copy the new files to `/etc/rancher/k3s/` and `/var/lib/rancher/k3s/server/manifests/`; add a `shell` provisioner step to `systemctl enable k3s.service amazon-ssm-agent`.
- `packer/scripts/install-k3s.sh` â€” bump `K3S_VERSION` default; remove the helm-repo-cache section (no longer needed at boot since chart fetch happens via helm-controller at runtime â€” though we may still want it for offline-tolerance; revisit).

- `variables.tf` (root) â€” remove `argocd_chart_version` (and propagate); optionally keep `app_of_apps_repo_url` as a Packer build input rather than a Terraform variable.

## Implementation sequence

1. **PR A: chart bumps in `eks-helm-charts`**
   - Bump `cert-manager/Chart.yaml`: `version`, `appVersion`, `dependencies[0].version` â†’ `v1.20.2`.
   - Bump `envoy-gateway/Chart.yaml`: `appVersion` â†’ `1.8.0`; `dependencies[0].version` â†’ `v1.8.0`.
   - Bump `external-dns/Chart.yaml`: `appVersion` and `dependencies[0].version` â†’ `1.21.1`.
   - Review release notes per chart for breaking changes; adjust template values if needed.
   - Merge to `main`.
   - Watch ArgoCD: all apps should reach `Synced + Healthy` on the 1.30 cluster within ~5 min. If not, fix before proceeding.

2. **PR B: declarative bootstrap + k3s 1.36 in `terraform`**
   - Add `packer/files/k3s-config.yaml`.
   - Add `packer/files/k3s-manifests/{00,10,20}-*.yaml`.
   - Update `packer/k3s-portfolio.pkr.hcl` to copy those files to the right paths during build and to `systemctl enable k3s.service amazon-ssm-agent`.
   - Bump `K3S_VERSION` in `packer/scripts/install-k3s.sh`.
   - Delete `tf-modules/aws-k3s/user_data.sh`.
   - Remove `user_data` + `user_data_replace_on_change` from `tf-modules/aws-k3s/main.tf`.
   - Remove `argocd_chart_version` and `app_of_apps_repo_url` Terraform variables (root + module).
   - Open PR. Do not merge yet.

3. **Build new AMI from PR B branch**
   - `gh workflow run build-ami.yml -f reason="k8s 1.36 + declarative bootstrap" --ref <PR-B-branch>`
   - Wait for completion (~6-8 min).
   - Verify new AMI tagged `Cluster=fuhriman-k3s` + `ManagedBy=Packer`.

4. **Merge PR B**

5. **`terraform apply` to replace the instance**
   - Plan should show: AMI ID change â†’ instance replacement. No `user_data` change in the diff (because it no longer exists).
   - Apply. Instance replacement starts; EIP automatically re-associates with the new instance.
   - Wait for boot + k3s + helm-controller + ArgoCD: ~3-5 min.

6. **Refresh local kubeconfig and SSM tunnel**
   - `terraform output -raw kubeconfig_retrieval_command | bash`
   - Restart the SSM port-forward tunnel.

7. **Wait for App-of-Apps reconciliation**
   - App-of-Apps Application installs immediately (no chart wrapper now â€” it's just a plain Application manifest deployed by helm-controller).
   - Child Applications follow sync waves: cert-manager (-2) â†’ envoy-gateway (-1) â†’ external-dns + fuhriman-website (0).
   - Full reconciliation: ~3-5 min.

## Validation criteria

All must pass before declaring upgrade complete:

- [ ] `kubectl get nodes` â†’ `STATUS=Ready`, `kubeletVersion: v1.36.1+k3s1`
- [ ] `kubectl get helmchart -A` â†’ `argocd` helm chart shows `JOBNAME=helm-install-argocd-<hash>`, completed
- [ ] `kubectl get application -n argocd` â†’ all 5 apps `SYNC=Synced`, `HEALTH=Healthy` at the post-merge commit SHA
- [ ] `kubectl get certificate -A` â†’ `fuhriman-tls` `READY=True`
- [ ] All HTTPRoutes show `Accepted=True / ResolvedRefs=True` on their parents
- [ ] `kubectl get pods -A --field-selector=status.phase!=Running,status.phase!=Succeeded` â†’ empty
- [ ] `curl -sI https://fuhriman.org`, `https://www.fuhriman.org`, `https://argocd.fuhriman.org` â†’ 200/302 with valid LE cert
- [ ] `terraform plan` against the deployed state shows zero diff (no drift from removed user_data)
- [ ] No `/var/log/k3s-init.log` (file should not exist â€” proves user_data didn't run)

## Rollback

| Scope | Action | Recovery time |
| --- | --- | --- |
| Chart bumps cause drift on live 1.30 cluster (after PR A merge) | `git revert` PR A in `eks-helm-charts`, push. ArgoCD re-syncs to old charts. | ~5 min |
| Packer build fails | Investigate; do not merge PR B. Cluster unaffected. | n/a |
| `terraform apply` fails before instance comes up | `terraform apply` again (idempotent). If unrecoverable, revert PR B + re-apply. | ~10 min |
| New instance boots, k3s starts, but helm-controller fails to install argocd | SSM into instance, `kubectl describe helmchart -n argocd argocd` for error. Common: bad chart version, missing image. Fix in Packer files, rebuild AMI. | ~15 min |
| New instance boots, argocd installs, but App-of-Apps Application doesn't reconcile | Check repoURL in baked manifest. If git URL wrong, rebuild AMI. If permissions, ArgoCD doesn't need creds for public repo. | ~10-15 min |
| New instance boots, ArgoCD healthy, but workloads broken on k8s 1.36 | Revert PR A in `eks-helm-charts`. ArgoCD on the new k3s syncs to old charts. If incompatible with 1.36, accept ~10 min outage and roll back PR B too. | ~10-25 min |
| Catastrophic | `terraform destroy` + `terraform apply` on reverted code in both repos. Full rebuild from git. | ~15 min |

## Risks

1. **k3s `HelmChart` CR behavior unfamiliar** â€” helm-controller is k3s-specific and less commonly debugged than `helm install`. Failure modes: ImagePullBackOff on the `klipper-helm` job, RBAC issues, manifest size limits. Mitigation: test the manifest manually on the live 1.30 cluster before merging PR B (kubectl apply of just `10-argocd-helm.yaml` should produce a working argocd install).
2. **`argocd-apps` 1.6 â†’ 2.0 major bump** â€” no longer applies (D7 eliminates it).
3. **cert-manager 1.14 â†’ 1.20** â€” six minors of webhook tightening and CRD evolution. Most relevant: ClusterIssuer schema, ACME challenge specs, gatewayHTTPRoute solver behavior.
4. **envoy-gateway 1.3 â†’ 1.8** â€” Gateway API graduated v1beta1 â†’ v1 in this window. The recent drift-loop fix (commit `30c946a` in `eks-helm-charts`) already applies the v1 schema explicitly, which should make this bump smoother.
5. **Bundled Gateway API CRDs version jump** â€” if envoy-gateway 1.8 bundles Gateway API v1.5.x CRDs and any existing live HTTPRoute/Gateway uses pre-v1 fields, conversion webhooks could fail mid-rollout. Mitigation: clean-slate AMI rebuild means no live state at the moment of CRD install.
6. **k3s manifest auto-deploy is order-dependent only by filename** â€” `00-`, `10-`, `20-` prefixes establish order (alphabetical). helm-controller doesn't block the next manifest on the previous one's completion; ArgoCD Application creation could race with argocd readiness. Mitigation: ArgoCD will retry the Application reconciliation once its server is up, so eventual consistency holds.
7. **No log file for first-boot debugging** â€” without `user_data.sh`, there's no `/var/log/k3s-init.log` to tail. First-boot diagnostics shift to `journalctl -u k3s` and `kubectl get helmchart,helmchartconfig -A`. Document this in CLAUDE.md.
8. **`amazon-ssm-agent` must be explicitly enabled at Packer time** â€” currently `user_data.sh` does `systemctl enable --now amazon-ssm-agent`. After PR B, Packer must do the equivalent during AMI build, else SSM access breaks on the new instance.
9. **kubeconfig staleness post-replace** â€” guaranteed (k3s regenerates its CA on new instance). Standard fix: `terraform output -raw kubeconfig_retrieval_command | bash`.

## Known unknowns

- Exact Gateway API CRD version envoy-gateway 1.8 bundles (need to check chart's `crds/` directory or templates at implementation time).
- Whether cert-manager 1.20 requires any opt-in feature gates for the `gatewayHTTPRoute` solver (1.14 needed `ExperimentalGatewayAPISupport=true`; later versions may have made it stable).
- Whether `helm-controller` in k3s 1.36 supports the same `HelmChart` CR schema as documented for k3s 1.30 â€” schema has been stable for years but worth a quick docs check.
- Whether removing the helm-repo cache from the AMI saves enough boot time to matter (currently the Packer scripts pre-populate `/root/.config/helm/repositories.yaml` for the runtime helm install â€” without runtime helm, this is dead weight).
