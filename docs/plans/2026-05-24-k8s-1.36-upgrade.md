# Design: Upgrade k3s cluster to Kubernetes 1.36

**Status:** Approved design, awaiting implementation plan.
**Date:** 2026-05-24
**Author:** Adam Fuhriman (with Claude)
**Scope:** AWS-side infrastructure (`terraform/`) and Helm charts (`eks-helm-charts/`).
**Sibling repo impact:** `eks-helm-charts` (chart bumps), `argocd-app-of-apps` (no changes expected), `fuhriman-website` (no changes expected).

---

## Context

The single-node k3s cluster on `t4g.medium` is currently running **k3s v1.30.4+k3s1** (Kubernetes 1.30), 6 minor versions behind the latest stable **k3s v1.36.1+k3s1** (Kubernetes 1.36). All in-cluster workloads are managed declaratively by ArgoCD via the App-of-Apps pattern, so cluster state is fully reconstructible from git. This makes a clean-slate upgrade strategy viable.

Cert budget check (2026-05-24): only 1 Let's Encrypt issuance in the past 41h vs the 5-per-168h duplicate-certificate limit. The AMI rebuild strategy is safe from rate-limit considerations for this window.

## Goals

1. Bring the cluster to Kubernetes 1.36 (k3s v1.36.1+k3s1).
2. Bring all Helm chart dependencies (`cert-manager`, `envoy-gateway`, `external-dns`, `argocd-apps`) to their current latest stable versions in the same change.
3. Preserve public availability of `fuhriman.org`, `www.fuhriman.org`, `argocd.fuhriman.org`. Downtime budget: **~10 minutes** during instance replacement.
4. Preserve git as the source of truth — no in-cluster mutations, no `kubectl edit` workarounds.

## Non-goals

- Changing the cluster topology (still single-node, single-AZ, t4g.medium).
- Migrating away from k3s, Packer, or the App-of-Apps pattern.
- Bumping `fuhriman-chart` (in-house chart, already at 1.0.0 and unaffected by k8s version).
- Bumping the ArgoCD Helm chart itself — the live install at `9.5.15` is already the latest stable.

## Decisions

### D1: Target k8s minor version → **1.36 (k3s v1.36.1+k3s1)**

Chosen over: v1.35.5+k3s1 (one back, more baked), v1.30.x patch (zero-risk no-progress). Rationale: a 6-minor jump and a 5-minor jump cost the same in chart compatibility work; might as well land on current.

### D2: Upgrade strategy → **AMI rebuild + instance replace**

Chosen over: in-place upgrade (~30s API blip, preserves state), blue/green (~$0.50 overlap cost, zero downtime). Rationale: the repo's plumbing (`user_data_replace_on_change = true`, App-of-Apps fully reconstructible state, Packer pipeline already wired through GitHub Actions OIDC) is purpose-built for this strategy. Cumulative etcd migrations across 6 minors in an in-place upgrade is the largest risk to avoid; clean-slate sidesteps it entirely.

### D3: Chart bump scope → **all charts to latest stable**

Chosen over: minimum-for-1.36-compat, minimum-now-sweep-later. Rationale: opportunistic maintenance sweep while we're in this state anyway. Avoids a follow-up cycle and gets every component to the same "modern" baseline.

### D4: Sequencing → **chart-bump PR before k3s-bump PR**

Smoke-test chart bumps against the live 1.30 cluster first (most chart bumps are backward-compatible). Catches chart issues without confounding them with k3s issues. If chart drift appears, fix in isolation before the AMI rebuild.

## Implementation sequence

1. **PR A: chart bumps in `eks-helm-charts`**
   - Bump `cert-manager/Chart.yaml`: `version`, `appVersion`, `dependencies[0].version` → `v1.20.2`.
   - Bump `envoy-gateway/Chart.yaml`: `appVersion` → `1.8.0`; `dependencies[0].version` → `v1.8.0`.
   - Bump `external-dns/Chart.yaml`: `appVersion` and `dependencies[0].version` → `1.21.1`.
   - Review release notes per chart for breaking changes; adjust template values if needed.
   - Merge to `main`.
   - Watch ArgoCD: all apps should reach `Synced + Healthy` on the 1.30 cluster within ~5 min. If not, fix before proceeding.

2. **PR B: k3s + argocd-apps bump in `terraform`**
   - Bump `packer/scripts/install-k3s.sh` `K3S_VERSION` default → `v1.36.1+k3s1`.
   - Bump `tf-modules/aws-k3s/user_data.sh` argocd-apps chart version reference → `2.0.5` (after reviewing 2.x release notes for breaking values changes).
   - Bump `variables.tf` `argocd_chart_version` default only if needed (currently `9.5.15` and already latest — leave).
   - Open PR. Do not merge yet.

3. **Build new AMI from PR B branch**
   - `gh workflow run build-ami.yml -f reason="k8s 1.36 upgrade" --ref <PR-B-branch>`
   - Wait for completion (~6-8 min).
   - Verify new AMI tagged `Cluster=fuhriman-k3s` + `ManagedBy=Packer` (so the Terraform AMI data source picks it).

4. **Merge PR B**

5. **`terraform apply` to replace the instance**
   - Plan should show: AMI ID change, `user_data` change (forces instance replacement), new instance ID.
   - Apply. Instance replacement starts; EIP automatically re-associates with the new instance (no DNS change needed).
   - Wait for cloud-init / user_data to complete (~3-5 min).

6. **Refresh local kubeconfig**
   - `terraform output -raw kubeconfig_retrieval_command | bash`
   - Restart the SSM port-forward tunnel (the old instance's session is dead).

7. **Wait for App-of-Apps reconciliation**
   - `app-of-apps` Application installs first (via `argocd-apps` chart in user_data.sh).
   - Child Applications follow sync waves: cert-manager (-2) → envoy-gateway (-1) → external-dns + fuhriman-website (0).
   - Full reconciliation: ~3-5 min.

## Validation criteria

All must pass before declaring upgrade complete:

- [ ] `kubectl get nodes` → `STATUS=Ready`, `kubeletVersion: v1.36.1+k3s1`
- [ ] `kubectl get application -n argocd` → all 5 apps `SYNC=Synced`, `HEALTH=Healthy` at the post-merge commit SHA
- [ ] `kubectl get certificate -A` → `fuhriman-tls` `READY=True`
- [ ] `kubectl get httproute -A -o jsonpath='{range .items[*]}{.metadata.name}: {.status.parents[0].conditions[?(@.type=="Accepted")].status} / {.status.parents[0].conditions[?(@.type=="ResolvedRefs")].status}{"\n"}{end}'` → all `True / True`
- [ ] `kubectl get pods -A --field-selector=status.phase!=Running,status.phase!=Succeeded` → empty
- [ ] `curl -sI https://fuhriman.org` → 200 with valid LE cert
- [ ] `curl -sI https://www.fuhriman.org` → 200 with valid LE cert
- [ ] `curl -sI https://argocd.fuhriman.org` → 200/302 with valid LE cert

## Rollback

| Scope | Action | Recovery time |
|---|---|---|
| Chart bumps cause drift on live 1.30 cluster (after PR A merge) | `git revert` PR A in `eks-helm-charts`, push. ArgoCD re-syncs to old charts. | ~5 min |
| Packer build fails | Investigate; do not merge PR B. Cluster unaffected. | n/a |
| `terraform apply` fails before instance comes up | `terraform apply` again (idempotent). If unrecoverable, revert PR B + re-apply. | ~10 min |
| New instance boots but k3s fails | Revert PR B in `terraform`, re-trigger Packer build for the old k3s version (or pick a recent previous AMI from the keep-3 retention), `terraform apply`. | ~15 min |
| New instance boots, k3s healthy, but workloads broken | Revert PR A in `eks-helm-charts`. ArgoCD on the new k3s syncs to old charts. If incompatible with 1.36, accept ~10 min outage and roll back PR B too. | ~10-25 min |
| Catastrophic | `terraform destroy` + `terraform apply` on reverted code in both repos. Full rebuild from git. | ~15 min |

## Risks

1. **`argocd-apps` 1.6 → 2.0 major bump** — likely a values schema rename. Must read release notes; might need user_data.sh values adjustment.
2. **cert-manager 1.14 → 1.20** — six minors of webhook tightening and CRD evolution. Most relevant: ClusterIssuer schema, ACME challenge specs, gatewayHTTPRoute solver behavior.
3. **envoy-gateway 1.3 → 1.8** — Gateway API graduated v1beta1 → v1 in this window. The recent drift-loop fix (commit `30c946a` in `eks-helm-charts`) already applies the v1 schema explicitly, which should make this bump smoother.
4. **Bundled Gateway API CRDs version jump** — if the envoy-gateway 1.8 chart bundles Gateway API v1.5.x CRDs and any existing live HTTPRoute/Gateway uses pre-v1 fields, conversion webhooks could fail mid-rollout. Mitigation: clean-slate AMI rebuild means no live state at the moment of CRD install.
5. **Bootstrap timing creep** — newer chart versions = larger images, more init containers. Could push user_data.sh runtime past current ~60s. Existing `until` loop on `get.k3s.io` is tolerant. Hard ceiling is cloud-init's default timeout (1h+), so this is comfort not correctness.
6. **kubeconfig staleness post-replace** — guaranteed (k3s regenerates its CA on new instance). Standard fix: `terraform output -raw kubeconfig_retrieval_command | bash`.
7. **SSM tunnel must be restarted** — old instance's session is gone with the instance.

## Known unknowns

- Exact Gateway API CRD version envoy-gateway 1.8 bundles (need to check chart's `crds/` directory or templates at implementation time).
- argocd-apps 2.x values schema changes (need to read release notes at implementation time).
- Whether cert-manager 1.20 requires any opt-in feature gates for the `gatewayHTTPRoute` solver (1.14 needed `ExperimentalGatewayAPISupport=true`; later versions may have made it stable).

## Open questions for plan stage

- Should PR A and PR B be opened simultaneously, or PR A merged before PR B is opened?
- Do we want to take a manual EBS snapshot before `terraform apply`, in addition to the monthly DLM policy? (Recoverability ladder: snapshot → AMI keep-3 → git revert.)
